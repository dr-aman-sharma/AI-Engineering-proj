# AI Coding Agent: Autonomous Software Engineering with Gemini Function Calling



This project implements a fully autonomous coding agent that leverages Google's Gemini API to perform complex software engineering tasks through natural language conversation. Unlike traditional code assistants that merely generate responses, this agent actively interacts with a codebase by exploring directory structures, reading file contents, writing modifications, and executing programs. The agent then analyzes the outputs of those executions to verify whether its changes produced the desired behavior, creating a complete feedback loop that enables genuine autonomous debugging and repair.

The architecture centers on four core functions that Gemini can call as needed. The getfilesinfo function lists all files and directories within a specified path, returning each item's size and whether it is a directory, allowing the agent to understand the project's structure before diving into specific files. The getfilecontent function reads the full text of any file within the working directory, with built-in truncation to prevent excessively large files from overwhelming the context window. The writefile function overwrites existing files or creates new ones, automatically generating any missing parent directories along the way. Finally, the runpythonfile function executes any Python script with optional command-line arguments, capturing both standard output and standard error while enforcing a thirty-second timeout to prevent hanging processes. Each function includes security checks that verify all file paths remain within the designated working directory, preventing the agent from accessing sensitive parts of the filesystem.

The agent's decision-making loop runs inside a maximum of twenty iterations, with each cycle sending the conversation history to Gemini along with the available function declarations. When Gemini determines that a function call would help answer the user's request, it returns a structured function call that the agent executes locally. The result of that function call is then packaged as a tool response and appended to the conversation history, allowing Gemini to see the outcome of its action and decide on the next step. This continues until Gemini produces a final text response instead of another function call, at which point the agent prints that response to the console. The agent also supports a verbose mode that displays token usage counts and prints the name of each function as it is called, providing transparency into the decision-making process.

To demonstrate the agent's capabilities, the project includes a deliberately broken calculator application. The calculator evaluates arithmetic expressions but initially contains a critical precedence bug where addition executes before multiplication, causing expressions like three plus seven times two to evaluate to twenty instead of the mathematically correct seventeen. When instructed to fix this bug, the agent begins by calling getfilesinfo to understand the project structure, then calls getfilecontent on the calculator's main evaluation logic and on the rendering module to understand how results are displayed. After identifying that the precedence values assign addition a higher priority than multiplication, the agent calls writefile to swap these values, making multiplication take precedence. The agent then suggests running the calculator again, and when the user does so manually, the expression correctly evaluates to seventeen. This complete debugging workflow, from the initial broken output through the agent's function calls to the final corrected result, is captured in screenshot "codingagent7.png" located in the screenshots folder.

The system prompt instructs Gemini to act as a helpful coding agent that should begin any investigation by looking at the project's files, figuring out how to run the project and its tests, and verifying that behavior matches expectations before attempting modifications. All paths provided in function calls must be relative to the working directory, which the agent's execution environment automatically injects for security purposes. This design ensures that the agent cannot escape the sandboxed directory even if Gemini generates malicious or malformed paths. The working directory is set to the calculator folder by default, keeping the agent focused on the demonstration application while still showcasing the full range of its capabilities.

This project demonstrates a practical implementation of tool-augmented language models, showing how an LLM can move beyond passive text generation to become an active agent that reads, writes, executes, and verifies code. The complete feedback loop of action and observation enables the agent to debug real software defects without any human intervention beyond the initial instruction. The calculator example, while simple, illustrates a fundamental capability that scales to larger codebases and more complex bugs. By providing the agent with the ability to run code and see the results, the system creates a tight iterative loop where the LLM can test hypotheses, observe failures, and refine its approach until success. All of these capabilities, from basic file reading to sophisticated autonomous debugging, are thoroughly documented in the screenshots folder, providing clear visual evidence of the agent's functionality across a wide range of tasks.
